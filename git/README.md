# 팀프로젝트에서 깃의 중요성

## 저장소
저장소란 소스코드가 저장되는 보관소이다.
### 원격 저장소
서버(클라우드)에 있는 공유 공간
- 깃, 클라우드 등등

### 로컬 저장소
- 내 컴퓨터에 있는 작업 공간

## Git 협업 워크플로우

![alt text](image.png)

### 1. 브랜치 생성
![alt text](image-1.png)
브랜치는 '메인코드와 분리되어' 코드를 실험하는 안전한 실험실이다
- 성공하면 합치고, 실패하면 삭제한다.
![alt text](image-2.png)

### 2. 커밋
![alt text](image-3.png)
커밋은 깃에 기록을 남기는 단위
- 누가, 언제, 뭘 했는지 보여줌
- 시간이 아니라 기능 단위로 커밋을 할 것!
- 시점 : 논리적으로 완결된 작업 단위
- 횟수 : 작은 단우로 자주 커밋
- 커밋 컨벤션 : 효율적인 팀 협업을 위한 필수 약속

## 3. Merge Request
![alt text](image-4.png)
함께 코드를 확인하는 것이다. Code Review 과정을 거치게 하기 위해 MR이 있다.
- Merge Review -> Code Review -> Approve

## 4. Merge branch
충돌은 에러가 아니다. "어떤 코드를 쓸까"라는 Git의 질문이다.
![alt text](image-5.png)


## 5. 미련 없이 삭제하기
- Branch는 Merge를 끝내면 삭제하자!
    - 혹시나 나중에 문제가 생길 거라는 걱정은 금물! 커밋 기록이 남아있다면 언제든 깃 커밋을 되돌릴 수 있다.

### 최소한 이건 꼭 지키자!
- main(master), develop 브랜치에 직접 커밋 금지
    - 무조건 브랜치를 만들어서 작업하고 MR을 통해서만 적용하기
- 브랜치 이름 작성 큐칙 준수
    - Sjh_work 이런 거 쓰지 말고, feature/기능명, fix/버그명 이 형식 지키기
- 작업 끝난 브랜치 즉시 삭제하기
    - Merge 끝난 브랜치는 바로 삭제하기. 쌓아두면 나중에 혼란만 가중됨.
- 1일 1커밋 & 잦은 Merge
    - 하루 최소 1커밋. 
    - 브랜치를 너무 오래유지 하지 마세요!
        - 충돌 가능성이 무지 커짐

    - 2 ~ 3일 안에 merge하는 것이 이상정
